## 2026-07-15 辅助功能权限与悬浮识别反馈

背景：
- ad-hoc 二进制更新后，macOS 辅助功能列表仍显示旧授权为开启，但新二进制读取不到权限；启动按钮又自动打开系统设置，形成循环。
- 菜单栏内的小预览无法让用户持续确认手掌是否被识别。

决策：
- 安装路径固定为 `/Applications/GestureControl.app`；替换二进制后只重置一次旧 TCC 记录，再对最终安装版本授权。
- 启动按钮不再自动打开设置；只有用户点击权限行的“设置”按钮才请求辅助功能。App 每秒刷新状态，授权生效后自动继续启动。
- 控制开启后默认弹出悬浮测试窗，显示镜像视频、手部骨架、捏合连线和实时状态。

原因：
- 权限状态必须由 App 实际读取结果决定，不能相信系统设置里同名旧条目的表面开关。
- 系统级输入没有网页叠层，测试阶段必须提供持续、可见的识别反馈。

影响：
- macOS 原型升级为 `0.1.1`；后续重新构建 ad-hoc App 时仍可能需要重置 TCC，正式签名后再消除该开发期限制。
- 悬浮窗只用于状态反馈，不接收手势输入，不改变前台 App 目标。

相关文件：
- `macos-app/Sources/GestureControlApp/AppModel.swift`
- `macos-app/Sources/GestureControlApp/DebugWindow.swift`
- `macos-app/scripts/install-app.sh`

## 2026-07-15 系统级 macOS 原型替代浏览器多页面重构

背景：
- 浏览器插件多页面不稳定来自 `activeTab`、MV3 生命周期、页面注入和标签页绑定，不是手势识别算法本身。
- 用户的真实目标是 MacBook 上跨应用阅读控制，而不是只控制 Chrome。

决策：
- 浏览器插件冻结在本地 `v1.0.1`，第 04 阶段改做原生 macOS 菜单栏 App。
- 第一版使用 AVFoundation + Apple Vision + Core Graphics，只实现左挥上翻、右挥下翻和捏住上下拖动滚动。
- 不移动系统光标，不做鼠标点击或 Air Mouse；系统事件只发给当前前台 App，切换 App 立即取消手势。

原因：
- 系统级事件消除标签页、网页权限和 Content Script 约束，能直接覆盖 Chrome、Safari、Preview、Notion 等场景。
- Air Mouse 会引入光标映射、抖动、误点和疲劳，不能与最小翻页原型混在一起验证。

影响：
- 新增独立 `macos-app/` 版本线，原型版本从 `0.1.0` 开始。
- 需要摄像头和辅助功能权限，不需要屏幕录制权限。
- 浏览器插件代码和历史版本完整保留，但暂不继续多页面优化。

相关文件：
- `macos-app/`
- `docs/production-blueprint.md`
- `docs/dev-log/archive/03-browser-extension-v1-pivot.md`

## 2026-07-14 1.0 打包与控制窗口停留规则

背景：
- 用户确认核心手势控制达到 1.0 可交付状态，但发现高级设置在摄像头未运行时消失，且点击网页会因窗口失焦过早收起。
- 产品需要同时面向 Chrome 与 Microsoft Edge（Chromium），并形成可上传商店的可复现包。

决策：
- 高级设置入口始终放在摄像头开关下方；展开后显示两条灵敏度滑杆与网格、捏合圆点两个开发者开关。
- 只以窗口 `pointerleave` 作为收起触发条件；删除 `blur` 收起路径。鼠标在主窗口范围内时，窗口不会消失。
- 固定 `npm run package:extension` 为 1.0 发布包命令，生成 manifest 位于 ZIP 根目录的 MV3 包；Chrome 与 Edge 共用同一构建产物。

原因：
- 高级设置是工具入口，不是摄像头运行状态；不该无故消失。
- 失焦并不等于用户离开控制窗口，尤其点击网页正是使用手势控制的常规动作。
- 本项目使用的 `offscreen`、`scripting`、`storage` 等 MV3 API 均在 Microsoft Edge 的官方支持列表中，适合用同一代码包作为两个商店的基础。

影响：
- 展开高级设置后即使摄像头关闭，也能预先调整并持久化参数；设置会在下次启动立即生效。
- ZIP 本身用于商店提交；本机开发安装仍必须选择解压后的 `dist-extension/`。

相关文件：
- `extension/sidepanel.html`
- `extension/sidepanel.ts`
- `scripts/package-extension.mjs`
- `README.md`

## 2026-07-14 网页真实反色、关闭恢复与可调灵敏度

背景：
- 用户实机确认捏合点在浅色和蓝色页面上仍显示为白色，说明 Shadow DOM 内的 `mix-blend-mode` 没有得到网页像素作为混合背景。
- 关闭预览窗口后，用户遇到网页不再响应手势；同时需要按个人习惯微调横挥与捏合门槛。

决策：
- 捏合点移出 Shadow DOM，作为网页 Light DOM 顶层节点用 `backdrop-filter: invert(1)` 直接反转它覆盖的网页画面。
- 原生控制窗口关闭时，Service Worker 读取后台状态、再次确认/注入网页接收器，再恢复入口黑点与识别状态；不把控制窗口生命周期当作摄像头生命周期。
- 新增本机持久化的 0–100 高级设置：挥手值越高，需要的横向位移越短；捏合值越高，允许的指尖间隙略大。默认 50 仍为历史阈值。

原因：
- 真正的 backdrop filter 才能反映图片、渐变和分区背景，而不是只猜某个 CSS 颜色。
- “预览关了，控制还在”必须由恢复链路显式保证，不能只依赖 Offscreen Document 理论上还活着。

影响：
- 新增 `storage` 权限，仅保存两项本机数值，不保存摄像头画面或手势轨迹。
- 展开高级设置时原生控制窗口高度增至 540px；收起后恢复紧凑尺寸。

相关文件：
- `extension/content-script.ts`
- `extension/service-worker.ts`

## 2026-07-14 反色捏合点与控制窗口收口

背景：
- 用户希望捏合点在浅色、深色网页上都清楚可见，且不要绿色、投影或比入口更大的视觉存在。
- 展开控制窗口时不应同时露出右侧入口黑点；启停控件也不应像播放器。

决策：
- 捏合点使用 28px 的白色 DOM 圆点与 `mix-blend-mode: difference`，让浏览器按其下方网页像素自动反色；不加投影。
- 入口也保持 28px，位置下移；控制窗口右缘贴齐 Chrome 内容区，并在展开态显式隐藏入口。
- 原生控制窗口收紧为 300×350，保留 16:9 视频，启停改为满宽圆角文字按钮“开启摄像头 / 关闭摄像头”。

原因：
- 反色是唯一不需要猜测网页背景色、又能保持极简的方案。
- 入口与控制点同尺度，窗口从入口展开的关系更干净，不再像两个孤立的黑块。

影响：
- `mix-blend-mode` 只影响网页内视觉层，不会改变网页内容或系统鼠标；捏合与滚动判定逻辑不变。

## 2026-07-14 严格捏合与贴边紧凑控制窗口

背景：
- 用户反馈此前捏合阈值过宽，食指和拇指尚未贴合时也会出现网页控制点；15 FPS 网页状态更新让点的移动可见跳帧。
- 用户希望把控制窗口压缩、移动到网页右侧小黑点附近，并有从小点展开、点击网页后收回的视觉关系。

决策：
- 捏合分类和捏合滚动均以归一化距离 `0.18` 为接触阈值；一旦超过阈值立即取消网页点和连续滚动。
- Offscreen 状态同步提升到 30 FPS；Content Script 使用动画帧插值显示 36px 不透明绿色点与极轻投影。
- 原生 Chrome 弹窗不能嵌入网页，因此控制窗口采用 300×400 的贴边原生窗口；Service Worker 依据 Chrome 窗口坐标把它放在右侧中部。网页小黑点在原生窗口开关时缩放淡入淡出，提供等效的展开/收起动效。

原因：
- “严格接触”是精细控制的显式离合器，能避免普通手势误进入滚动模式。
- 状态频率与页面渲染频率分层后，视觉更平滑而不需要提高模型推理或上传任何数据。

影响：
- 实际原生窗口仍由 Chrome 管理，无法与网页 DOM 完全合并或逐帧跟随；网页圆点承担连续性动画。

相关文件：

## 2026-07-14 网页极简反馈与横挥上下滚动

背景：
- 用户不需要持续的手部网格或黑色动作面板，只需要明确知道“已捏合”和“本次会向哪里滚动”。
- 原来的左右挥动翻页难以在普通网页稳定验证，且和捏合滚动不是同一套阅读模型。

决策：
- 网页仅在捏合时显示无特效的半透明绿色中点；松开、丢手或识别过期立即清空。
- 左挥映射上滚并显示中央空心 `↑`，右挥映射下滚并显示中央空心 `↓`；不再发送合成方向键。
- 网页入口缩为右侧 28px 黑色圆点；控制窗口删除动作屏，只保留摄像头预览与启停。

原因：
- 反馈与用户的实际意图一一对应，同时不占据阅读区域。
- 右挥下翻和捏合上拖下翻形成两条可直接对比的“向下浏览”路径。

影响：
- 左右翻页能力从本轮产品范围移除；未来若需要站点级翻页，应以单独适配器重新引入。

## 2026-07-14 网页接收器以确认式注入恢复

背景：
- 用户实机截图显示 `Could not establish connection. Receiving end does not exist.`；摄像头骨架正常，说明问题发生在后台到网页的最后一跳。
- 旧实现只要 `executeScript()` Promise 返回就认为网页控制脚本可用，且构建产物带有运行时模块依赖；这不够可靠。

决策：
- 网页 Content Script 改为无运行时 import 的独立文件。
- 注入后立即 ping 确认接收器已建立；动作或网页反馈若发现接收器消失，自动补注入后只重试一次。

原因：
- 动作链路必须以“网页确实收到消息”为成功条件，而不是以“尝试注入没有报错”为成功条件。

影响：
- 页面跳转、扩展重载后可自行恢复；正常捏合拖动不再每帧 ping，避免消息检查影响滚动手感。

相关文件：
- `extension/content-script.ts`
- `extension/service-worker.ts`

## 2026-07-14 网页消息按接收方严格过滤

背景：
- 加入网页手部网格后，Offscreen Document 会持续发布识别状态。
- Chrome 会将扩展 runtime 消息送达 Content Script；旧代码没有先确认消息属于网页协议，可能把后台状态或捏合请求落进通用网页动作分支，导致页面不滚动。

决策：
- Content Script 只接受 `ContentScriptRequest` 白名单中的六种消息；其他扩展内部消息直接忽略。

原因：
- 后台识别、Service Worker 路由和网页执行是三个独立接收方，消息类型必须明确隔离，不能依赖发送方“刚好不被收到”。

影响：
- 手部网格、黄色黄点和捏合/左右动作可同时运行；背景识别消息不再有机会干扰网页控制。

相关文件：
- `extension/message-types.ts`
- `extension/content-script.ts`

## 2026-07-14 反向捏合拖动与网页手部指针

背景：
- 用户认为手向上移动的意图是查看下方内容；此前实现按同向滚动，导致感知与结果相反。
- 视频控制窗口关闭后，用户无法确认后台模型看到的手在网页空间中的位置。

决策：
- 捏合拖动采用触控板/触屏的直接操作模型：手向上拖动时文档内容向上移动，页面滚动向下，从而显示下方内容。
- 后台识别每秒最多 15 次把手部状态转发到受控网页；网页以镜像坐标绘制 21 点手部网格，捏合时在拇指尖与食指尖中点绘制黄色球。

原因：
- 内容与手的移动方向一致，符合用户“把页面往上推”的直觉。
- 黄色球是明确的空间锚点，能区分“模型没看到手”和“捏合还没接管滚动”。

影响：
- 网页叠层仅在本机渲染，不上传画面或关键点；全屏画布不接收指针事件，不阻挡网页点击。
- 黄色球的坐标是摄像头画面到当前网页视口的镜像映射，不是系统级鼠标控制。

## 2026-07-14 网页方形提示成为控制入口

背景：
- 用户认为插件图标更像“重新启动”，而运行中的网页方形提示才是手势控制的自然入口。
- 旧控制窗口下方使用四方向按钮阵列，重复表达了动作状态，视觉过重。

决策：
- 点击网页内 88×88 方形提示，直接打开关联标签页的摄像头控制窗口；插件图标保留作第一次启动和备用入口。
- 控制窗口改成视频预览、单一黑色动作屏和一个启动/停止按钮；动作屏只在识别时闪一个绿色箭头。

原因：
- 入口跟随正在运行的功能，用户不必回到浏览器工具栏寻找插件。
- 一个即时动作屏比四个常驻箭头更轻、更像仪表，而不是控制台。

影响：
- 网页提示改为可点击和键盘可访问；点击只打开预览，不会重新启动后台识别。
- 控制窗口从 390×720 收紧为 340×560。

## 2026-07-14 删除冗余紧凑控制窗口

背景：
- 用户点击网页后，独立控制窗口会缩成一个黑色窄标签；网页本身又有一个黑色方形动作提示，造成两个黑块。
- Offscreen Document 已独立持有摄像头和手势识别，不需要紧凑窗口继续保活。

决策：
- 删除紧凑控制标签及其箭头反馈。
- 鼠标离开控制窗口后直接关闭它，只保留网页内的 88×88 方形提示；用户需要停止摄像头时重新点击插件图标打开控制窗口。

原因：
- 一个明确的网页内状态点已经够了；第二个黑块没有提供额外信息，只会遮挡内容。

影响：
- 关闭控制窗口会停止其预览流，但不会停止后台识别、捏合滚动或左右翻页。

## 2026-07-14 捏合连续滚动替代上下挥动

背景：
- 上下挥动没有稳定的身体参照，用户实机测试中容易把方向或动作本身做错。
- 当前 HandLandmarker 已经输出拇指尖和食指尖的 21 点关键点，工程也已有归一化捏合距离计算。

决策：
- 上下滚动改为拇指和食指捏住后上下拖动：稳定接触约 80ms 才启动，松开即停止；手向上移动时页面向上。
- 保留张开手掌的左右挥动翻页；Extension 禁用上下挥动，以免两个输入模式抢同一条垂直轨迹。
- 控制窗口在两指接触时，将拇指尖与食指尖的中点绘制成黄色球。

原因：
- 捏合是显式的“我要控制”信号，连续拖动更接近鼠标滚轮/触控板而非四个模糊的空中方向。
- 黄色球让用户确认识别的是手势接触点，不是猜模型有没有看见手。

影响：
- 新增 `PinchScrollDetector`，并覆盖启动、拖动、松开/跳变及握拳保护的单元测试。
- 无需新模型、权限或网络请求；仍基于本机 `HandLandmarker` 的拇指尖（4）和食指尖（8）关键点。

## 2026-07-12 手势与性能方案

背景：
- 用户需要位置转向、张手继续、握拳暂停，并增加手掌左右旋转控制。
- MediaPipe 同步运行在主线程时，真实浏览器实测最低约 19 FPS。
- 3DXRguy 的 VIVERSE 源码调研显示其使用 `HandLandmarker`、GPU 优先、逐点平滑和短暂丢失续帧。

决策：
- 使用 `HandLandmarker` float16 模型和 21 个关键点，不使用 `GestureRecognizer` 内置分类。
- 手腕到中指根部的屏幕平面角度映射为虫洞旋转。
- 手掌倾斜控制的是旋转速度而非固定角度：保持倾斜会持续旋转，回正后停止。
- 张手、握拳和捏合由关节距离计算，收拇指不影响张手判断。
- 前进速度由手掌在画面中的相对尺寸控制：首次识别校准为 1×，靠近摄像头更快、远离更慢，并限制在 0.45×～2.4×。
- MediaPipe WASM 推理放入 ES Module Web Worker，自适应 24–30 FPS，GPU 优先、CPU 备用。
- 关键点逐点平滑；检测短暂失败时保持最后可信姿态 220ms。

原因：
- 关节距离判断不依赖拇指和内置标签；Worker 避免阻塞 Three.js 渲染。

影响：
- 摄像头开启后的黑洞渲染恢复约 60 FPS；模型和 WASM 从本项目同源加载。
- 极端 90° 侧面仍受单目 RGB 摄像头物理限制，但短时遮挡和拇指收拢的体验会明显改善。

## 2026-07-14 用控制窗口替代 Side Panel

背景：
- 用户在 macOS Chrome 的 Side Panel 中启动摄像头，Chrome 网页权限和系统权限均正常，但 `getUserMedia()` 仍返回 `NotAllowedError`。
- Chromium 扩展支持渠道已确认：Side Panel 没有摄像头/麦克风授权弹窗所需的交互界面。

决策：
- 不再由 Chrome Side Panel 承载摄像头。
- 用户从目标网页点击插件图标后，Service Worker 注入网页控制脚本，并打开独立的控制窗口；该窗口在用户点击“启动摄像头”时请求权限。
- 控制窗口 URL 带原始 tabId，后续挥动始终控制触发它的网页，而不是控制窗口自身。

原因：
- 这是 Chrome Side Panel 在 macOS 上的产品限制，继续调权限没有意义。
- 控制窗口保留常驻体验，同时拥有授权摄像头的正常交互环境。

影响：
- 移除 `sidePanel` 权限；用户不再从 Chrome 侧栏菜单打开插件。
- 版本升级到 `0.3.0`，需要用户在 `chrome://extensions/` 重新加载解压扩展。
